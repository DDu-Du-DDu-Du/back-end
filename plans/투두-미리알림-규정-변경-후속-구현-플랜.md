# 투두 미리알림 규정 변경 후속 구현 플랜

## 1. 목표

- PR 본문에 정의된 신규 스펙에 맞춰 투두의 단건 미리알림(`remindAt`) 중심 구조를 제거하고, Reminder 다건 구조를 기준으로 `GET /api/todos/{id}` 및 `POST /api/todos` 동작을 일치시킨다.
- 기존 투두 미리알림 전용 API(`PATCH /api/todos/{id}/reminder`, `DELETE /api/todos/{id}/reminder`)와 연계 유스케이스/도메인 로직을 제거한다.
- `Todo`에서 발생시키던 미리알림 Interim 이벤트 처리 책임을 `reminder` 패키지 서비스로 이동해 경계를 명확하게 한다.
- DB 스키마에서 `todos.remind_at`를 제거하고 Flyway 마이그레이션 및 `afterMigrate.sql`을 동기화한다.

## 3. 단계별 구현 순서

1. **도메인 정리**
   - `Todo` aggregate 및 관련 command 객체에서 reminder 단건 로직 제거.
   - Reminder 시간 검증 주체를 Reminder 도메인으로 고정하고, 검증 입력은 `Todo.getScheduleDatetime` 기준으로 전달.
2. **API 계약 변경**
   - `CreateTodoRequest`, `TodoDetailResponse`, 관련 UseCase 시그니처 업데이트.
3. **유스케이스 재구성**
   - `CreateTodoService`에서 reminder 목록 생성/검증을 reminder out 포트 + 도메인 로직으로 수행.
   - `RetrieveTodoService`에서 reminder 목록 조회를 reminder out 포트로 조합.
   - `UpdateTodoService`의 reminder 처리 유무를 확인하고, 생성 시나리오와 동일한 규칙으로 반영.
4. **레거시 제거**
   - set/cancel reminder 유스케이스, 서비스, 컨트롤러 엔드포인트, 문서 삭제.
5. **인프라/DB 정합성 반영**
   - `TodoEntity`와 Flyway/afterMigrate 동기화.
6. **이벤트 책임 이동**
   - Interim 이벤트 발행/처리 주체를 reminder 패키지 서비스로 이동.

## 5. 리스크 및 확인 포인트

- Reminder 생성 시 투두 일정(`scheduledOn`, `beginAt`) 기준으로 과거 시각 입력 검증을 어디에서 수행할지(도메인 vs 서비스) 책임을 명확히 해야 한다.
- `RetrieveTodoService`에서 Reminder 조회 시 정렬 기준(`remindsAt ASC`)이 고정되어야 클라이언트 UI가 안정적이다.
- 기존 앱/웹이 호출하던 `PATCH/DELETE /api/todos/{id}/reminder` 제거에 따른 하위호환 공지/배포 순서가 필요하다.
- Flyway 반영 이후 로컬/CI 테스트 데이터 스크립트(`afterMigrate.sql`) 불일치 여부를 반드시 확인한다.

## 6. 삭제 대상 메서드 사용처 영향 분석 및 대응

- 분석 대상(`Todo`): `validateReminder`, `resolveReminder`, `isReminderInputEmpty`, `setReminder`, `cancelReminder`, `getReminderDifference`, `hasReminder`
- 사용처 점검 범위
  - `application/planning-application`의 `CreateTodoService`, `UpdateTodoService`, `SetReminderService`, `CancelReminderService`
  - `bootstrap/planning-api`의 `TodoController`, `TodoControllerDoc`
  - `infra/planning-infra-mysql`의 `TodoEntity`, `TodoPersistenceAdapter`
  - `application/notification-application`, `bootstrap/notification-*`, `domain/notification-domain`의 Todo 연계 참조 여부
  - 테스트 코드(`TodoTest`, 서비스 테스트) 내 직접 호출/검증 포인트
- 사전 점검 절차
  1. 제거 대상 메서드의 전역 참조를 수집하고 호출 경로를 문서화
  2. Notification 모듈에서 Todo 단건 reminder 메서드/필드를 직접 참조하는지 별도 확인
     - 현재 기준으로 Notification 모듈 main 소스에서 `Todo.hasReminder/setReminder/cancelReminder/getReminderDifference/getRemindAt` 직접 참조는 없음
  3. 레거시 API 제거 범위(PATCH/DELETE)와 함께 삭제 가능한 경로를 분리
  4. 생성/수정/조회 경로에서 대체 로직(포트+Reminder 도메인) 적용 여부를 체크리스트화
- 이슈 발생 시 방안
  - **컴파일 이슈**: 제거 메서드 참조부를 우선 포트/도메인 대체 경로로 전환 후, 레거시 코드 삭제를 후순위로 분리 커밋
  - **동작 이슈**: reminder 검증 실패/정렬 불일치 발생 시, `Todo.getScheduleDatetime` 입력값과 reminder out 포트 결과를 로그/테스트로 교차 검증
  - **회귀 이슈**: `CreateTodoServiceTest`, `RetrieveTodoServiceTest`, `UpdateTodoServiceTest`에 회귀 케이스를 추가해 재발 방지
  - **Notification 연계 이슈**: Notification 모듈에서 Todo 정보를 기반으로 reminder 판단이 필요해지는 경우, `todoId`로 Todo를 조회한 뒤 `Todo.getScheduleDatetime`을 입력값으로 Reminder 도메인 로직(검증/계산)을 호출하는 경로로 통일
