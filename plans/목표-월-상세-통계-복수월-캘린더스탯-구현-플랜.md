# 목표 월 상세 통계 복수월 캘린더 스탯 구현 플랜

## 1) 구현 목표

- `GET /api/stats/detail/{goalId}/achieved`, `GET /api/stats/detail/{goalId}/postponed` 응답의 `calendarStats`에서 `isAvailable` 필드를 제거한다.
- 단일월/복수월 여부와 관계없이 요청 월 범위(`fromMonth` ~ `toMonth`)에 포함되는 월별 캘린더 통계를 반환한다.
- `calendarStats`는 월 단위 배열로 변경하고, 각 원소는 `yearMonth`와 해당 월의 일별 통계 `stats`를 가진다.
- 기존 상세 통계의 `overview`, `dayOfWeekStats`, `repeatTodoStats`, `goalId`, `goalColor` 응답 의미는 유지한다.

### 목표 응답 예시

```json
{
  "goalId": 1,
  "goalColor": "BLUE",
  "overview": {},
  "dayOfWeekStats": {},
  "repeatTodoStats": [],
  "calendarStats": [
    {
      "yearMonth": "2026-05",
      "stats": [
        {
          "date": "2026-05-06",
          "totalCount": 0,
          "completedCount": 0,
          "uncompletedCount": 0
        }
      ]
    }
  ]
}
```

## 2) 현행 코드 기준 변경 방향

- 현재 상세 통계 응답은 `GenericCalendarStats<TodoCompletionResponse>`를 사용하며 `isAvailable`과 `stats`를 함께 노출한다.
- 현재 `CollectMonthlyStatsDetailService#getCalendarCompletions(...)`는 `toMonth`가 `fromMonth`보다 이후이면 `GenericCalendarStats.from(false, Collections.emptyList())`를 반환해 복수월 캘린더 통계를 차단한다.
- 변경 후에는 복수월 차단 분기를 제거하고, 요청 범위의 월마다 `TodoCompletionResponse` 목록을 조회/분류해 `calendarStats` 배열을 생성한다.
- 달성 중심 캘린더 통계는 기존처럼 `scheduledOn` 기준으로 집계한다.
- 미루기 중심 캘린더 통계는 기존 미루기 상세 통계 정책과 동일하게 `postponedAt` 기준으로 조회하되, 응답의 일자(`date`)는 현재 `TodoCompletionResponse`가 가진 의미와의 정합성을 별도로 점검한다.

## 3) 신규/변경 패키지 및 클래스

### A. 응답 DTO 패키지

- **변경 패키지**
  - `application/application-common/src/main/java/com/modoo/application/common/dto/stats`
- **신규 클래스 후보**
  - `MonthlyCalendarStats<T>`
    - 역할: 월별 캘린더 통계 응답 단위.
    - 필드 후보:
      - `YearMonth yearMonth`
      - `List<T> stats`
    - 정적 팩토리 후보:
      - `from(YearMonth yearMonth, List<T> stats)`
- **변경 클래스 후보**
  - `GenericCalendarStats<T>`
    - 선택지 1: 더 이상 사용처가 없으면 삭제한다.
    - 선택지 2: 월별 캘린더 응답 DTO로 재정의한다.
    - 권장: 기존 이름은 `isAvailable` 포함 구조를 연상시키므로, 신규 `MonthlyCalendarStats<T>`를 추가하고 기존 `GenericCalendarStats<T>`는 사용처 제거 후 삭제한다.

### B. 상세 통계 응답 패키지

- **변경 패키지**
  - `application/application-common/src/main/java/com/modoo/application/common/dto/stats/response`
- **변경 클래스**
  - `AchievedStatsDetailResponse`
    - `calendarStats` 타입 변경:
      - 기존: `GenericCalendarStats<TodoCompletionResponse>`
      - 변경: `List<MonthlyCalendarStats<TodoCompletionResponse>>`
  - `PostponedStatsDetailResponse`
    - `calendarStats` 타입 변경:
      - 기존: `GenericCalendarStats<TodoCompletionResponse>`
      - 변경: `List<MonthlyCalendarStats<TodoCompletionResponse>>`
  - `TodoCompletionResponse`
    - 기본 필드는 유지한다.
    - 단, 미루기 중심 조회에서 `postponedAt` 기준 날짜로 그룹핑할 경우 `date`가 `postponedAt.toLocalDate()`를 의미해야 하는지 검증하고, 필요 시 쿼리 projection을 보완한다.

### C. 유즈케이스 서비스 패키지

- **변경 패키지**
  - `application/stats-application/src/main/java/com/modoo/application/stats/service`
- **변경 클래스**
  - `CollectMonthlyStatsDetailService`
    - `getCalendarCompletions(...)`의 복수월 불가 분기를 제거한다.
    - 반환 타입을 `List<MonthlyCalendarStats<TodoCompletionResponse>>`로 변경한다.
    - `fromMonth`부터 `toMonth`까지 월 단위로 순회하며 각 월의 시작일/종료일을 계산한다.
    - 각 월별로 `todoStatsPort.calculateTodosCompletion(...)`을 호출하거나, 포트가 범위 전체를 한 번에 조회하도록 확장한 뒤 서비스에서 월별로 그룹핑한다.
    - 응답 빌더의 `calendarStats(...)`에는 월별 DTO 목록을 주입한다.

### D. 출력 포트 패키지

- **변경 가능 패키지**
  - `application/application-common/src/main/java/com/modoo/application/common/port/stats/out`
- **변경 가능 클래스**
  - `TodoStatsPort`
    - 기존 메서드 `calculateTodosCompletion(...)`은 월/주간 완료율 API에서도 사용되므로 호환성을 고려한다.
    - 선택지 1: 기존 메서드를 유지하고 서비스에서 월별 반복 호출한다.
    - 선택지 2: 상세 통계 전용 신규 메서드를 추가한다.
      - 예: `calculateTodosCompletionByMonths(LocalDate startDate, LocalDate endDate, Long userId, Long goalId, List<PrivacyType> privacyTypes, boolean isAchieved)`
    - 권장: 우선 변경 범위가 작은 선택지 1을 적용하고, 성능 이슈가 확인되면 선택지 2로 확장한다.

## 4) 단계별 구현 계획

1. **월별 캘린더 DTO 추가**
   - `MonthlyCalendarStats<T>`를 추가한다.
   - `yearMonth`, `stats` 필드를 정의하고 필요 시 정적 팩토리를 제공한다.

2. **상세 응답 DTO 타입 변경**
   - `AchievedStatsDetailResponse.calendarStats`를 `List<MonthlyCalendarStats<TodoCompletionResponse>>`로 변경한다.
   - `PostponedStatsDetailResponse.calendarStats`를 동일하게 변경한다.
   - 기존 `GenericCalendarStats` import를 제거한다.

3. **서비스 캘린더 조립 로직 변경**
   - `CollectMonthlyStatsDetailService#getCalendarCompletions(...)` 반환 타입을 변경한다.
   - `toMonth.isAfter(fromMonth)`일 때 빈/불가 응답을 반환하는 분기를 삭제한다.
   - `fromMonth`부터 `toMonth`까지 순회하는 private helper를 추가한다.
   - 각 월별 시작일/종료일로 `todoStatsPort.calculateTodosCompletion(...)`을 호출해 월별 DTO를 만든다.
   - `calendarStats` 응답 순서는 `fromMonth`부터 `toMonth`까지 오름차순으로 고정한다.

4. **미루기 캘린더 날짜 기준 점검 및 필요 시 쿼리 보정**
   - 현재 미루기 조회는 `postponedAt` 범위 조건을 사용하지만, projection/group by/order by는 `scheduledOn`을 사용한다.
   - 요구사항상 미루기 캘린더의 날짜가 미룬 날짜를 의미해야 한다면 `TodoQueryRepositoryImpl#findTodosCompletion(...)`에서 `isAchieved=false`일 때 `postponedAt.toLocalDate()` 기준으로 group by/order by/projection을 분기한다.
   - Querydsl/JPA에서 날짜 변환이 복잡해질 경우 상세 통계 전용 조회 메서드를 분리한다.

5. **API 문서 갱신**
   - 상세 통계 API 문서의 예시 JSON에서 `isAvailable`을 제거한다.
   - `calendarStats`를 배열 구조로 바꾸고 `yearMonth`, `stats` 예시를 추가한다.

6. **테스트 수정/추가**
   - 기존 `isAvailable` 성공/실패 케이스를 삭제하거나 신규 구조 검증으로 변경한다.
   - 달성 중심 복수월 캘린더 통계 성공 케이스를 추가한다.
   - 미루기 중심 복수월 캘린더 통계 성공 케이스를 추가한다.
   - 데이터 없는 월 포함 케이스를 추가해 월 단위 DTO가 누락되지 않는지 확인한다.

7. **정리 작업**
   - `GenericCalendarStats` 사용처가 사라지면 클래스를 삭제한다.
   - 사용하지 않는 import와 `Collections.emptyList()` 분기용 import를 제거한다.
   - 포맷 및 Checkstyle 위반 여부를 확인한다.

## 6) 리스크 및 확인 포인트

- `YearMonth` 직렬화 형식이 목표 예시처럼 `yyyy-MM`인지 확인한다.
- 미루기 중심 캘린더 통계의 `date` 기준이 `scheduledOn`인지 `postponedAt`인지 제품 요구사항과 최종 확인한다.
- 복수월 범위가 길어질 경우 월별 반복 조회가 N번 발생할 수 있으므로, 성능 요구사항에 따라 범위 일괄 조회 후 애플리케이션에서 그룹핑하는 방식으로 확장한다.
- `calendarStats` 타입 변경은 API 응답 스키마의 breaking change이므로 프론트엔드 반영 일정과 함께 배포한다.
- `GenericCalendarStats` 삭제 시 다른 모듈에서 사용하는지 `rg "GenericCalendarStats"`로 재확인한다.

## 7) 작업 완료 정의(DoD)

- 상세 통계 응답 2종에서 `calendarStats.isAvailable`이 더 이상 노출되지 않는다.
- 단일월과 복수월 상세 통계 요청 모두 `calendarStats[].yearMonth`와 `calendarStats[].stats`를 반환한다.
- 달성 중심/미루기 중심 애플리케이션 서비스 테스트가 신규 응답 구조 기준으로 통과한다.
- 관련 API 문서 예시가 신규 응답 구조와 일치한다.
