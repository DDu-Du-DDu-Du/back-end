# 뚜두 날짜변경 미루기 버그픽스 구현 플랜

## 1) 배경 및 목표

- 현행: `PUT /api/ddudus/{id}/date` 호출 시 `postpone=true`이면 `postponedAt`이 **미루기 요청 시각(now)** 으로 저장된다.
- 목표: `postpone=true`이면 `postponedAt`에 **변경 전 기존 scheduledOn(기존 일자)** 이 저장되도록 수정한다.
- 비기능 목표: 기존 완료 뚜두 미루기 실패 규칙(`UNABLE_TO_POSTPONE_COMPLETED_DDUDU`)과 기존 API 계약은 유지한다.

## 2) 신규/변경 영향 클래스 및 패키지

### 2.1 Domain

1. 변경 대상 클래스
   - `domain/planning-domain/src/main/java/com/ddudu/domain/planning/ddudu/aggregate/Ddudu.java`
2. 패키지
   - `com.ddudu.domain.planning.ddudu.aggregate`
3. 변경 포인트
   - `moveDate(LocalDate newDate, boolean postpone)` 내부 `postponedAt` 계산 로직 수정
   - 기존: `LocalDateTime.now()`
   - 변경: 변경 전 `this.scheduledOn` 기반 시각값(예: `this.scheduledOn.atStartOfDay()`) 사용

### 2.2 Application (UseCase Layer)

1. 영향 클래스(동작 확인/테스트 중심)
   - `application/planning-application/src/main/java/com/ddudu/application/planning/ddudu/service/MoveDateService.java`
2. 패키지
   - `com.ddudu.application.planning.ddudu.service`
3. 확인 포인트
   - 서비스는 현재처럼 `Ddudu.moveDate(request.newDate(), request.postpone())` 호출만 유지
   - 실제 버그 수정은 도메인 내부에서 해결되므로 서비스 시그니처/흐름은 변경하지 않음

### 2.3 API / DTO (계약 확인)

1. 영향 클래스(계약 회귀 확인)

   - `com.ddudu.api.planning.ddudu.controller`
3. 확인 포인트
   - 입력 파라미터(`newDate`, `postpone`) 및 API 엔드포인트 변경 없음

### 2.4 Infra / Persistence (매핑 확인)

1. 영향 클래스(저장 정합성 확인)

## 3) 상세 구현 단계

1. 도메인 수정
   - `Ddudu.moveDate(newDate, true)`에서 기존 `scheduledOn`을 로컬 변수로 캡처
   - `postponedAt`에 캡처한 기존 일자 기반 값을 세팅
   - `scheduledOn`은 `newDate`로 변경

2. 도메인 단위 테스트 보강
   - 성공 케이스: 미루기 시
     - `scheduledOn == newDate`
     - `postponedAt`은 변경 전 일정일자 기반 값
   - 실패 케이스: 완료된 뚜두에 미루기 요청 시 기존 예외 유지

3. 애플리케이션 서비스 테스트 보강
   - `MoveDateServiceTest`에서 저장 후 조회한 `postponedAt` 검증 추가
   - `postpone=false`일 때 `postponedAt`이 의도치 않게 설정되지 않는지 회귀 검증

4. 회귀 검증
   - `domain/planning-domain` 및 `application/planning-application` 테스트 실행
   - 필요 시 미루기 통계/조회에 영향 없는지 smoke 확인 (`postponedAt` not null 조건은 유지)

## 4) 테스트 계획

1. Domain 테스트
   - `DduduTest`의 날짜 변경 테스트 섹션에서 `postponedAt` 기대값 명시 검증

2. Application 테스트
   - `MoveDateServiceTest`에서 `moveDate(..., postpone=true)` 결과의 `postponedAt` 검증

3. 실행 순서
   - (DB DDL 변경 없음으로 생략 가능)
   - `./gradlew :domain:planning-domain:test --tests "*DduduTest"`
   - `./gradlew :application:planning-application:test --tests "*MoveDateServiceTest"`

## 5) 리스크 및 확인 사항

- `postponedAt`의 시간부(`HH:mm:ss`)를 무엇으로 볼지 합의 필요
  - 권장: `scheduledOn.atStartOfDay()`로 고정하여 날짜 의미를 명확히 유지
- 과거 데이터는 이미 now 기반으로 저장되었을 수 있으나, 본 이슈 범위는 신규 변경 로직 중심
- 통계/조회는 `postponedAt is not null` 여부를 사용하므로 기능적 영향은 제한적
