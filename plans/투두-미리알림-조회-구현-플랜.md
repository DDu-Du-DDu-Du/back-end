# 투두 미리알림 조회 구현 플랜 (Reminder 패키지 기준)

## 1) 목표

- `POST /api/reminders`로 생성된 미리알림 데이터를 동일한 bounded context(`planning.reminder`)에서 조회할 수 있도록 `GET /api/reminders?todoId={todoId}`를 구현한다.
- 조회 유스케이스의 중심 패키지를 `planning.todo`가 아닌 `planning.reminder`로 구성해 생성/조회 책임을 일관되게 유지한다.
- 기존 인증/인가 및 예외 정책(로그인 사용자 검증, 투두 소유자 검증, NotFound 처리)을 동일하게 따른다.

## 2) 구현 범위

### 2-1. Application Common

#### 신규 패키지/클래스

- `application/application-common/src/main/java/com/ddudu/application/common/dto/reminder/response/RetrieveReminderResponse.java`
  - 필드: `id`, `remindsAt`, `remindedAt`
- `application/application-common/src/main/java/com/ddudu/application/common/port/reminder/in/RetrieveRemindersUseCase.java`
  - 시그니처 예시: `List<RetrieveReminderResponse> retrieve(Long loginId, Long todoId)`

#### 변경 패키지/클래스

- `application/application-common/src/main/java/com/ddudu/application/common/port/reminder/out/ReminderLoaderPort.java`
  - todoId 기반 조회 메서드 추가(생성 API에서 저장한 reminder를 그대로 조회)

### 2-2. Planning Application (Reminder)

#### 신규 패키지/클래스

- `application/planning-application/src/main/java/com/ddudu/application/planning/reminder/service/RetrieveRemindersService.java`
  - `RetrieveRemindersUseCase` 구현
  - 처리 순서
    1) 로그인 사용자 조회
    2) 투두 조회
    3) 투두 소유자 검증
    4) `ReminderLoaderPort`로 `todoId` 기준 목록 조회
    5) `RetrieveReminderResponse` 매핑 후 반환

#### 신규 테스트 클래스

- `application/planning-application/src/test/java/com/ddudu/application/planning/reminder/service/RetrieveRemindersServiceTest.java`
  - 성공: 미리알림 조회 성공
  - 실패: 로그인 사용자 없음, 투두 없음, 타 사용자 투두 접근

### 2-3. Planning Infra (MySQL)

#### 변경 패키지/클래스

- `infra/planning-infra-mysql/src/main/java/com/ddudu/infra/mysql/planning/reminder/repository/ReminderRepository.java`
  - `todoId`(필요 시 `userId` 포함) 기준 목록 조회 메서드 추가
- `infra/planning-infra-mysql/src/main/java/com/ddudu/infra/mysql/planning/reminder/adapter/ReminderPersistenceAdapter.java`
  - `ReminderLoaderPort` 조회 메서드 구현

> 핵심: 조회도 `notification-event`가 아니라 `POST /api/reminders`가 쓰는 `planning.reminder` 저장소를 직접 사용한다.

### 2-4. Bootstrap Planning API

#### 변경 패키지/클래스

  - 기존 생성 엔드포인트(`POST /api/reminders`)와 같은 컨트롤러에서 조회 엔드포인트 추가

- `bootstrap/planning-api/src/main/java/com/ddudu/api/planning/reminder/doc/ReminderControllerDoc.java`
  - 조회 API 문서/응답/에러 예시 추가

## 3) 상세 작업 순서

1. `application-common`에 reminder 조회 DTO/UseCase/LoaderPort 계약을 추가한다.
2. `planning-application.reminder`에 `RetrieveRemindersService`를 구현한다.
3. `planning-infra-mysql.reminder` repository/adapter에 조회 메서드를 확장한다.
4. `ReminderController`/`ReminderControllerDoc`에 `GET /api/reminders`를 추가한다.
5. `RetrieveRemindersServiceTest`를 작성해 성공/실패 시나리오를 검증한다.

## 4) POST /api/reminders와의 연결 설계

- 생성(`POST /api/reminders`)과 조회(`GET /api/reminders`) 모두 `planning.reminder` 패키지를 사용한다.
- 생성 시 저장되는 aggregate/entity(`Reminder`)를 조회 시 동일한 loader port로 조회한다.
- 따라서 생성/조회 간 저장소 불일치(예: 생성은 reminder, 조회는 notification-event)를 제거할 수 있다.

## 5) 테스트 계획

- Application 서비스 통합 테스트(`@SpringBootTest`, `@Transactional`)로 검증
- `@BeforeEach`에서 사용자/투두/미리알림 데이터 준비
- 테스트 메서드명은 한국어
- `//given`, `//when`, `//then` 명시
- 실패 케이스는 `ThrowingCallable` 사용

## 6) 주의사항

- 조회 정렬 기준(`remindsAt` 오름차순 등)은 클라이언트 요구사항에 맞춰 repository 레벨에서 고정한다.
- 응답은 배열을 유지해 다건 reminder 확장성에 대비한다.
- 권한 검증(`todo.validateTodoCreator`)을 생략하지 않는다.
- 문서에는 404(사용자/투두), 403(권한), 401(인증) 예시를 명시한다.
